## 1. I2C Serial Interface Overview

The MCP45HVX1 communicates exclusively via I²C (2-wire serial interface). It is a client (slave) device only.
[Source: DS20005304B, p. 48]

Supported speeds:
- Standard mode (100 kHz)
- Fast mode (400 kHz)
- High-Speed mode (1.7 MHz)

All three modes are supported. [Source: DS20005304B, p. 48]

The VL supply powers the I²C digital logic. [Source: DS20005304B, p. 48]

---

## 2. I2C Signal Definitions and Bus States

**SCL** — Serial Clock. Generated by I²C host (master).  
**SDA** — Serial Data. Bidirectional, open-drain.  
[Source: DS20005304B, p. 48]

### Bus condition definitions

| Condition | Definition |
|---|---|
| **Start (S)** | SDA transitions High-to-Low while SCL is High |
| **Stop (P)** | SDA transitions Low-to-High while SCL is High |
| **Repeated Start (Sr)** | Start condition generated without preceding Stop; used to switch direction (host→client→host) without releasing bus |
| **Acknowledge (A)** | Receiving device pulls SDA Low during 9th SCL pulse |
| **Not Acknowledge (Ā)** | SDA remains High during 9th SCL pulse; used to signal error or end of transfer by receiver |
| **Data Bit** | SDA must be stable while SCL is High; transitions only when SCL is Low |

[Source: DS20005304B, p. 48–50]

The MCP45HVX1 generates A/Ā bits (as client/slave). The host generates the final Ā at the end of a read sequence. [Source: DS20005304B, p. 61]

## 3. Device Addressing

### 7-bit address format

```
Bit:  7    6    5    4    3    2    1    0
      1    0    1    1    1    A1   A0   R/W
      [Fixed = 0b1011 1]   [Var]   [R/W]
```

The fixed portion is `1011 1` (binary) — i.e., the upper 5 bits are `10111`.  
A1 and A0 correspond to the physical address pins (see `02_pinout_and_signals.md`).  
R/W = 0 for Write, R/W = 1 for Read.

Up to 4 devices may share one I²C bus (A1:A0 = 00, 01, 10, 11).

**Note:** The fixed address bits differ from the MCP44XX/MCP45XX/MCP46XX family which uses `0101 1x`. The MCP45HVX1 uses `1011 1`.  
[Source: DS20005304B, p. 51]

10-bit addressing is **not supported**. [Source: DS20005304B, p. 51]

### Control byte (I²C address byte)

In the datasheet figures, the control byte is shown as:

```
S | 1 0 1 1 | 1 A1 A0 | R/W | A
```

where:
- `1011` = fixed address bits [7:4]
- `1 A1 A0` = fixed `1` plus two address-pin bits [3:2:1]  
  *(Note: bit 3 of the 7-bit I2C address is always 1)*
- `R/W` = direction bit (0 = host writes; 1 = host reads)
- `A` = Acknowledge from MCP45HVX1

[Source: DS20005304B, p. 51, p. 59, p. 61]

## 4. High-Speed (HS) Mode

### Entering HS mode

The host sends an HS Master Mode Code (HSMMC) byte immediately after a Start condition:

```
HSMMC = 0000 1XXX  (binary)
```

The MCP45HVX1 does **not** Acknowledge the HSMMC byte; it detects it and internally switches to HS mode. The SCL clock frequency can then be increased to up to 1.7 MHz. [Source: DS20005304B, p. 52]

### HS mode operation

After HSMMC, standard I²C framing (Start, address, data, Stop) continues at HS speed.  
The device returns to Fast/Standard mode upon Stop condition. [Source: DS20005304B, p. 52–53]

### Spike suppressor / pulse gobbler

In HS mode, the MCP45HVX1 activates its internal spike (pulse) gobbler circuit. This circuit suppresses noise spikes shorter than 10 ns on the SCL/SDA lines to prevent false triggering. [Source: DS20005304B, p. 52]

### Current-source pull-up

HS mode uses a high-current source (SDAH/SCLH) pull-up topology, but the MCP45HVX1 pin functions remain SCL and SDA regardless of mode. [Source: DS20005304B, p. 52]

## 5. General Call (GC) Protocol

### GC address

The General Call address is `0x00` (7-bit address `0000000`, R/W = 0).  
[Source: DS20005304B, p. 53]

### GCEN bit

A **GCEN (General Call Enable)** bit controls whether the device responds to the General Call address. When GCEN = 1 (enabled), the device responds to GC commands. When GCEN = 0, GC commands are ignored.  

> **Gap note:** The register location and bit position of GCEN within the device is not fully specified in the extracted text. It appears in discussions of General Call behavior but is not shown with a register address or bit number in the available pages. See `08_variant_differences_and_open_questions.md`.

[Source: DS20005304B, p. 53–54]

### GC command formats

Two GC command frame sizes are defined:

#### 3-byte GC format (Write Wiper or Write TCON)

```
Byte 1: S | 0000 0000 | A          ← General Call address
Byte 2: GC Control byte            ← Defines operation
Byte 3: Data byte                  ← 8-bit data
        P
```

Valid GC Control byte values for 3-byte format:

| GC Control Byte (binary) | Operation |
|---|---|
| `1000 00x` | Write Wiper 0 (x = don't care) |
| `1100 00x` | Write TCON Register (x = don't care) |

[Source: DS20005304B, p. 53–54]

#### 2-byte GC format (Increment or Decrement Wiper)

```
Byte 1: S | 0000 0000 | A          ← General Call address
Byte 2: GC Control byte            ← Defines operation
        P
```

Valid GC Control byte values for 2-byte format:

| GC Control Byte (binary) | Operation |
|---|---|
| `1000 01x` | Increment Wiper 0 |
| `1000 10x` | Decrement Wiper 0 |

[Source: DS20005304B, p. 53–54]

### GC constraints

- Only **one** GC command is executed per GC control byte. Additional commands beyond the first in a single GC sequence are ignored or NACKed. [Source: DS20005304B, p. 54]
- The GC increment/decrement applies to the volatile wiper only, same restrictions as individual INC/DEC commands. [Source: DS20005304B, p. 54]
- Wiper updates from GC commands are subject to WLAT: if WLAT is High, the wiper value is latched and not updated. [Source: DS20005304B, p. 54]

### GC use case

The GC mechanism allows all MCP45HVX1 devices on a bus to update wiper values simultaneously (analogous to LDAC on DACs), eliminating sequential update delays. If only a subset of devices should respond to GC, the remaining devices must be placed on a separate I²C bus. [Source: DS20005304B, p. 72]

## 6. Command Byte Format

The MCP45HVX1 command byte is always the second byte in a write sequence (after the I²C control byte + Acknowledge). It contains three fields:

```
Bit:  7    6    5    4    3    2    1    0
     AD3  AD2  AD1  AD0   C1   C0   D9   D8
     [Memory Address  ]  [Cmd] [Compat bits]
```

| Field | Bits | Description |
|---|---|---|
| AD3:AD0 | [7:4] | Memory address being accessed |
| C1:C0 | [3:2] | Command operation |
| D9:D8 | [1:0] | Unused/reserved; set to `xx` for compatibility with MCP44XX/MCP45XX/MCP46XX |

### Command codes (C1:C0)

| C1 | C0 | Command |
|---|---|---|
| 0 | 0 | Write Data |
| 0 | 1 | Increment Wiper |
| 1 | 0 | Decrement Wiper |
| 1 | 1 | Read Data |

[Source: DS20005304B, p. 56]

D9 and D8 are called "compatibility bits." They maintain command format compatibility with MCP44XX/MCP45XX/MCP46XX. D8 is mentioned as "currently only one of the data bits is defined (D8)," but its use/definition is not further described in the extracted pages. [Source: DS20005304B, p. 56]

If AD3:AD0 and C1:C0 form an **invalid combination**, the device generates a Not Acknowledge and the host must issue a Start condition to reset the I²C state machine. [Source: DS20005304B, p. 56]

## 7. Memory Map and Supported Commands

**TABLE 7-2** from the datasheet: [Source: DS20005304B, p. 57]

| Address | Register | Supported Commands |
|---|---|---|
| 00h | Volatile Wiper 0 | Write Data, Read Data, Increment, Decrement |
| 01h–03h | Reserved | None (NACK + error condition on any command) |
| 04h | Volatile TCON0 Register | Write Data, Read Data only |
| 05h–FFh | Reserved | None |

Notes from TABLE 7-2:
1. Data memory is 8 bits wide; the two MSbs (D9:D8) are **ignored** by the device. They exist only for compatibility with MCP44XX/MCP45XX/MCP46XX command formats.
2. Increment and Decrement commands are **invalid** on address 04h (TCON) — sending them generates a NACK.
3. Read returns **two bytes**: MSB (first byte returned) is always `0x00`; actual 8-bit data is in the LSB (second byte). This 2-byte read format is for compatibility with MCP44XX/MCP45XX/MCP46XX.

[Source: DS20005304B, p. 57]

---

## 8. Data Byte

For Write and Read commands, data is transferred in a 10-bit-compatible format (two bytes), but the device register is 8 bits wide. The upper 2 bits (D9:D8) in the command byte carry the data MSBs; the following data byte carries D7:D0. Since D9:D8 are always ignored (mapped as reserved/unused), effectively only D7:D0 matter.

Read always returns **2 bytes**:
- Byte 1 (MSB): `0x00` (always, by design, for MCP44XX/MCP45XX/MCP46XX compatibility)
- Byte 2 (LSB): The actual 8-bit register value

[Source: DS20005304B, p. 58, p. 61]

## 9. Error Conditions and Bus Recovery

### NACK on invalid command

If the address bits (AD3:AD0) and command bits (C1:C0) form an invalid combination, the device generates a NACK. Any subsequent commands after an error are **ignored** until the bus is reset with a Start condition. [Source: DS20005304B, p. 58]

### Aborting a transmission

A Restart (Sr) or Stop (P) condition in the expected **data bit position** will abort the current command. Data will **not** be written to the device on a mid-transfer abort. [Source: DS20005304B, p. 58]

### Falling off the bus (passive high)

If the device receives an incomplete or invalid I²C command, it enters a passive High condition on SDA and ignores all signals until the next valid Start + control byte. [Source: DS20005304B, p. 61]

### Software Reset Sequence

To ensure the device's I²C state machine is in a known state:

```
S | 1 1 1 1 1 1 1 1 1 | S | P
Start — nine '1' bits — Start — Stop
```

- The **first Start** resets a device that is in receive-data mode.
- The **nine '1' bits** force reset if the device is holding SDA Low (driving an Acknowledge or transmitting a '0' data bit during a Read).
- The **second Start** prevents a spurious write if the host was reset mid-write.
- The **Stop** terminates the current activity.

This technique **only resets the I²C state machine**. It does **not** modify register contents. [Source: DS20005304B, p. 66]

> Referenced application note: AN1028. [Source: DS20005304B, p. 66]

## 10. Write Data Command

### Purpose
Write an 8-bit value to a valid volatile register (address 00h or 04h).

### Single write frame (Figure 7-2)

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 0 0 | x x | [wait A]   ← Command byte: addr + Write(00) + D9:D8(xx)
Host:   D7 D6 D5 D4 D3 D2 D1 D0 | [wait A]       ← Data byte
Host:    P  (or Sr for continuous)
```

Symbols: S = Start, A = Acknowledge (from device), P = Stop, x = don't care.

- Total bit clocks for single write: **29** [Source: DS20005304B, p. 58]
- The volatile register is updated **after** the complete byte transfer and Acknowledge.
- If Stop or Restart is generated **before** the Acknowledge, data is **not** written. [Source: DS20005304B, p. 59]

### Continuous write (Figure 7-3)

Multiple write operations without issuing Stop or repeating the control byte:

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 0 0 | x x | [wait A]
Host:   D7...D0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 0 0 | x x | [wait A]   ← Next command (no re-sending control byte)
Host:   D7...D0 | [wait A]
  ... (repeat as needed)
Host:    P
```

- Total bit clocks for continuous write: **18n + 11** (n = number of write repetitions) [Source: DS20005304B, p. 58]
- Writes **do not** need to target the same address in a continuous sequence.
- Continuous writes are valid only to volatile registers (00h and 04h).
- The note in Figure 7-3 states: "Only functions when writing the volatile wiper registers (AD3:AD0 = 00h and 01h) or the TCON registers (AD3:AD0 = 04h)." Note that 01h is listed in the figure note but TABLE 7-2 marks it as Reserved; this is a potential contradiction. See `08_variant_differences_and_open_questions.md`. [Source: DS20005304B, p. 60]

## 11. Read Data Command

### Purpose
Read the current value of a volatile register.

The memory address read is the **last address from the most recent valid write command byte**. If no write has occurred since POR/BOR, the address defaults to **00h**. [Source: DS20005304B, p. 61]

### Read from last address (Figure 7-4)

```
Host:    S | 1 0 1 1 1 A1 A0 | 1 | [wait A]       ← Control byte with R/W=1
Device: 0 0 0 0 0 0 0 0 | [Host sends A]           ← MSB byte (always 0x00)
Device: D7 D6 D5 D4 D3 D2 D1 D0 | [Host sends Ā]  ← LSB byte (actual data)
Host:    P  (or Sr)
```

- Total bit clocks for single read: **29** [Source: DS20005304B, p. 58]
- The host generates the Acknowledge (A) after the MSB byte to request the LSB.
- The host generates a Not Acknowledge (Ā) after the LSB byte to terminate.
- If Ā is received by the device (instead of A) after the MSB, the device aborts and releases the bus. [Source: DS20005304B, p. 61]
- The device does **not** corrupt the Device Memory Address after Repeated Start or Stop conditions. [Source: DS20005304B, p. 61]

### Random read (read from any address) — Figure 7-5

A Repeated Start is required to set the desired address and then read:

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]           ← Write direction
Host:   AD3 AD2 AD1 AD0 | 1 1 | x x | [wait A]        ← Command byte with C1:C0=11 (Read)
Host:    Sr                                              ← Repeated Start (no Stop)
Host:    1 0 1 1 1 A1 A0 | 1 | [wait A]                ← Read direction
Device: 0 0 0 0 0 0 0 0 | [Host sends A]               ← MSB byte (always 0x00)
Device: D7 D6 D5 D4 D3 D2 D1 D0 | [Host sends Ā]      ← LSB byte
Host:    P
```

- Total bit clocks for random read: **40** [Source: DS20005304B, p. 58]

### Continuous reads (Figure 7-6)

```
Host:    S | 1 0 1 1 1 A1 A0 | 1 | [wait A]
Device: 0x00 | [Host sends A]
Device: D7...D0 | [Host sends A]         ← Host ACKs to continue reading
Device: 0x00 | [Host sends A]
Device: D7...D0 | [Host sends A]
Device: 0x00 | [Host sends A]
Device: D7...D0 | [Host sends Ā]         ← Host NACKs to terminate
Host:    P
```

- Total bit clocks for continuous read: **18n + 11** [Source: DS20005304B, p. 58]
- Continuous reads are possible to all memory locations. [Source: DS20005304B, p. 61]

## 12. Increment Wiper Command

### Purpose
Increment the volatile wiper by 1 in a single command with minimal framing overhead.

### Valid on: Address 00h only (Volatile Wiper 0)

Applying Increment to any other address generates a NACK. [Source: DS20005304B, p. 63]

### Single increment frame (Figure 7-7)

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 0 1 | x x | [wait A]      ← Command byte: C1:C0 = 01
Host:    P  (or Sr to chain next command)
```

- Total bit clocks for single increment: **20** [Source: DS20005304B, p. 58]

### Continuous increments

Multiple INC commands can be chained without re-sending the control byte:

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 0 1 | x x | [wait A]      ← INC (n→n+1)
Host:   AD3 AD2 AD1 AD0 | 0 1 | x x | [wait A]      ← INC (n+1→n+2)
  ...
Host:    P
```

- Total bit clocks for continuous increment: **9n + 11** [Source: DS20005304B, p. 58]
- The command sequence can transition from INC to any other valid command without issuing a Stop. [Source: DS20005304B, p. 63]

### Increment boundary behavior

| Current Setting | 7-bit (MCP45HV31) | 8-bit (MCP45HV51) | INC operates? |
|---|---|---|---|
| Full Scale | 7Fh | FFh | **No** — clamped, no further increment |
| Any mid value | 40h–7Eh | 80h–FEh | Yes |
| Mid Scale | 3Fh | 7Fh | Yes |
| Any mid value | 01h–3Eh | 01h–7Eh | Yes |
| Zero Scale | 00h | 00h | Yes |

Maximum count: 100h (8-bit) or 80h (7-bit). At maximum, further INC commands are ignored (value not incremented). [Source: DS20005304B, p. 63]

### Wiper update timing
The wiper value transitions after each command Acknowledge when accessing volatile wiper registers. [Source: DS20005304B, p. 63]

## 13. Decrement Wiper Command

### Purpose
Decrement the volatile wiper by 1 in a single command with minimal framing overhead.

### Valid on: Address 00h only (Volatile Wiper 0)

Applying Decrement to any other address generates a NACK. [Source: DS20005304B, p. 64]

### Single decrement frame (Figure 7-8)

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 1 0 | x x | [wait A]      ← Command byte: C1:C0 = 10
Host:    P  (or Sr to chain next command)
```

- Total bit clocks for single decrement: **20** [Source: DS20005304B, p. 58]

### Continuous decrements

```
Host:    S | 1 0 1 1 1 A1 A0 | 0 | [wait A]
Host:   AD3 AD2 AD1 AD0 | 1 0 | x x | [wait A]      ← DEC (n→n-1)
Host:   AD3 AD2 AD1 AD0 | 1 0 | x x | [wait A]      ← DEC (n-1→n-2)
  ...
Host:    P
```

- Total bit clocks for continuous decrement: **9n + 11** [Source: DS20005304B, p. 58]
- The command sequence can transition from DEC to any other valid command without issuing a Stop. [Source: DS20005304B, p. 64]

### Decrement boundary behavior

| Current Setting | 7-bit (MCP45HV31) | 8-bit (MCP45HV51) | DEC operates? |
|---|---|---|---|
| Full Scale | 7Fh | FFh | Yes |
| Any mid value | 40h–7Eh | 80h–FEh | Yes |
| Mid Scale | 3Fh | 7Fh | Yes |
| Any mid value | 01h–3Eh | 01h–7Eh | Yes |
| Zero Scale | 00h | 00h | **No** — clamped, no further decrement |

Minimum count: 000h. At minimum, further DEC commands are ignored. [Source: DS20005304B, p. 64]

### Wiper update timing
The wiper value transitions after each command Acknowledge when accessing volatile wiper registers. [Source: DS20005304B, p. 64]

## 15. WLAT Pin Interaction with Commands

The WLAT (Wiper Latch) pin inhibits wiper updates when High. When Low, wiper updates proceed normally. The WLAT pin affects when the physical wiper moves, not when data is written to the register.

- If WLAT = High during a Write/INC/DEC: the command is received and Acknowledged, but the wiper position does **not** change until WLAT goes Low.
- WLAT transition timing: the WLAT signal must be stable for at least **10 ns before** or **200 ns after** the ACK rising edge. Transitions within this window produce indeterminate behavior. [Source: DS20005304B, p. 27, p. 50]

> See also `06_modes_interrupts_status_and_faults.md` § WLAT for full timing table.
